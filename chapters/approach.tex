% --- Chapter Introduction ---
This chapter outlines the architectural design of the Security Information and Event Management (SIEM) system, the implementation methodology, the specific tools selected, and the validation experiments performed to evaluate its effectiveness. The primary objective is to engineer a robust, scalable security monitoring solution explicitly tailored to meet the HAIT+ (Healthcare Accreditation Information Technology) cybersecurity guidelines for hospital environments.

\subsection{Data Collection Layer (The Endpoint and Agent)}
This foundational layer is responsible for gathering security telemetry directly from the hospital's infrastructure.
\begin{itemize}
\item \textbf{Endpoint Generation:} The host operating systems or specific applications generate security events. These are written to local logs (such as \texttt{/var/log/auth.log} or \texttt{/var/log/dpkg.log}) or manifest as system state changes.
\item \textbf{Agent-based Collection (The Observer):} Lightweight Wazuh agents deployed on endpoints (Ubuntu 24.04 and Windows 11) operate as background services. Using the \texttt{} configuration block, agents continuously monitor targeted directories in real-time. For custom HAIT+ metrics, agents are configured to execute local shell commands (e.g., \texttt{df -P} or \texttt{uptime}) and capture the standard output.
\end{itemize}

\subsection{Secure Transmission Layer}
To maintain the integrity of hospital data, the Wazuh Agent compresses and encrypts the collected log payloads using AES encryption. The agent transmits this data over a secure TCP connection (Port 1514) to the central manager. If network connectivity drops, the agent queues logs locally and transmits them once the connection is restored, ensuring zero data loss.

\subsection{Processing and Analysis Layer (The Brain)}
The central server is the core of the SIEM. It processes the raw, encrypted data feeds from the agents.
\begin{itemize}
\item \textbf{Wazuh Server (wazuh-analysisd):} The daemon decrypts incoming logs and processes them in three phases: Pre-decoding (extracting timestamps and hostnames), Decoding (identifying log types), and Rule Matching. The log is evaluated against custom HAIT+ XML rules. If it matches a threshold, a security alert is generated.
\item \textbf{Wazuh Indexer:} Processed data is sent to the Elasticsearch-powered indexer, making it highly searchable for rapid threat hunting.
\end{itemize}

\subsection{Presentation and Alerting Layer}
\begin{itemize}
\item \textbf{Wazuh Dashboard:} A centralized web interface providing a holistic view of the hospital's security posture, enabling IT administrators to monitor real-time HAIT+ compliance events and investigate incidents.
\end{itemize}

% --- Section 3.2: Implementation Methodology ---
\section{Implementation Methodology}
\label{sec:dev-methodology}
The project implementation moved beyond baseline installation to focus on advanced SIEM customization, addressing the specific operational requirements of a healthcare environment.

\begin{enumerate}
\item \textbf{Baseline Deployment:} The core Wazuh Server was deployed alongside virtualized endpoint agents (Ubuntu 24.04 Server representing a medical database, and Windows 11 representing a staff workstation).
\item \textbf{The "Command-to-Rule" Pipeline:} Because out-of-the-box SIEMs do not natively cover all HAIT+ guidelines (such as disk space exhaustion or CPU anomalies), a custom pipeline was engineered. Localized scripts were assigned to agents to check specific metrics. The output was ingested as generic logs, and custom RegEx rules were authored on the Manager to intercept and score these anomalies.
\item \textbf{Overcoming Technical Constraints:} During implementation, several engineering challenges were addressed:
\begin{itemize}
\item \textbf{Decoder Collision:} Built-in decoders misclassified critical MariaDB \texttt{[ERROR]} logs as generic Apache web logs. This was resolved by implementing Rule Hierarchy (Parent-Child rules) to intercept the misclassified logs, scan for database-specific keywords, and elevate them to Critical Alerts.
\item \textbf{Unreliable Windows Event IDs:} To counteract Microsoft's frequent deprecation of Event IDs during system updates, firewall monitoring was shifted to Command-Line Auditing (Event 4688). Monitoring the actual executed commands (e.g., \texttt{netsh advfirewall}) provided a version-agnostic security alert.
\end{itemize}
\end{enumerate}


% --- Section 3.3: Tools and Technologies ---
\section{Tools and Technologies}
\label{sec:tools}
The project leverages a combination of enterprise-grade and open-source software to build a highly capable, cost-effective security monitoring architecture.
\begin{itemize}
\item \textbf{SIEM Platform (Wazuh):} Chosen for its open-source flexibility, lack of licensing costs, and extensible XML-based rule engine.
\item \textbf{Endpoint Operating Systems:}
\begin{itemize}
\item \textbf{Ubuntu 24.04 LTS Server:} Used to host the medical database (MariaDB) and test Linux-based HAIT+ compliance indicators.
\item \textbf{Windows 11:} Used to represent user workstations, testing native Antivirus integration (Windows Defender) and Process Creation auditing.
\end{itemize}
\end{itemize}

% --- Section 3.4: System Validation Experiments ---
\section{System Validation Experiments}
\label{sec:evaluationPlan}
To validate the functionality of the custom SIEM architecture, a series of experimental case studies was conducted. Each test targeted a specific HAIT+ indicator to ensure the pipeline (Endpoint → Agent → Server) operated correctly.

\begin{itemize}
\item \textbf{Experiment 1: System Package Integrity (HAIT+ \#12):} To detect unauthorized patches, an \texttt{apt install} command was executed on the Ubuntu agent. The system successfully monitored \texttt{/var/log/dpkg.log}, parsing the install keyword and triggering custom Rule 100006.
\item \textbf{Experiment 2: Database Reliability (HAIT+ \#16):} Simulated database crash logs were injected into the MariaDB error log. The system successfully bypassed the decoder collision utilizing Rule Hierarchy to generate a Level 9 alert.
\item \textbf{Experiment 3: Resource Exhaustion (HAIT+ \#2, \#3):} Custom commands (\texttt{df -P} and \texttt{uptime}) were passed through the Command-to-Rule pipeline to successfully detect artificial spikes in CPU load and storage capacity.
\item \textbf{Experiment 4: Reconnaissance Detection (HAIT+ \#14):} File Integrity Monitoring (FIM) was tested by attempting unauthorized access to the \texttt{/etc/shadow} file, generating a critical alert mapped to the MITRE ATT\&CK framework (T1003.008).
\end{itemize}
For each experiment, the timely generation of the correct custom alert on the Wazuh Dashboard confirmed the success of the implementation.
