\section{Test Environment Configuration}
\label{sec:test-env-setup}
The final validation was conducted in a stable, virtualized network environment representing a district hospital's IT infrastructure. The deployed environment consisted of:
\begin{enumerate}
\item A \textbf{Wazuh Manager VM} acting as the central SIEM server.
\item An \textbf{Ubuntu 24.04 LTS VM} acting as the hospital's core medical database (MariaDB installed) with a Wazuh agent.
\item A \textbf{Windows 11 VM} acting as a hospital staff workstation with a Wazuh agent.
\end{enumerate}
Both agents successfully enrolled, maintained an "active" status, and securely transmitted telemetry to the central manager.

\subsection{Experiment 1: System Package Integrity (HAIT+ \#12)}
\label{sec:res-pkg-integrity}
\begin{itemize}
\item \textbf{Objective:} Validate the pipeline's ability to detect unauthorized software modifications.
\item \textbf{Execution:} The command \texttt{sudo apt install nmap} was executed on the Ubuntu agent.
\item \textbf{Result:} \textbf{Success.} The Wazuh agent successfully monitored the local \texttt{/var/log/dpkg.log} file in real-time. The central server decoded the payload and matched the \texttt{install} keyword against custom Rule 100006. As shown in Figure \ref{fig:hait12_list}, the system instantly triggered a Level 7 alert on the dashboard. Furthermore, Figure \ref{fig:hait12_details} demonstrates the system's ability to extract deep forensic context, including the exact command and the user who executed it.
\end{itemize}

\subsection{Experiment 2: Database Reliability \& Decoder Override (HAIT+ \#16)}
\label{sec:res-db-reliability}
\begin{itemize}
\item \textbf{Objective:} Validate the detection of critical database errors while overcoming the default Decoder Collision (MariaDB logs misclassified as Apache logs).
\item \textbf{Execution:} A critical database corruption message (\texttt{[ERROR] MariaDB: Table crashed}) was injected into the endpoint's logs.
\item \textbf{Result:} \textbf{Success.} The implemented Rule Hierarchy (Parent-Child rules) successfully hijacked the misclassified log. Custom Rule 100008 identified the database-specific keywords, overrode the default Level 0 classification, and escalated the event to a Level 9 Critical Alert, ensuring database failures are strictly monitored.
\end{itemize}

\subsection{Experiment 3: Resource Exhaustion Pipeline (HAIT+ \#2 \& \#3)}
\label{sec:res-resource}
\begin{itemize}
\item \textbf{Objective:} Validate the Command-to-Rule pipeline for system resource anomalies not natively tracked by default SIEM decoders.
\item \textbf{Execution:} The Ubuntu agent was configured to execute the \texttt{df \-P} shell command. A script was used to artificially fill the root partition past the 20% test threshold.
\item \textbf{Result:} \textbf{Success.} The standard output of the command was securely ingested as a generic OSSEC log. The manager successfully parsed the capacity percentage and triggered the custom HAIT+ \#2 Disk Space Anomaly warning.
\end{itemize}

\subsection{Experiment 4: Windows Command-Line Auditing (HAIT+ \#11)}
\label{sec:res-windows-audit}
\begin{itemize}
\item \textbf{Objective:} Detect unauthorized security modifications on Windows endpoints without relying on unstable Windows Event IDs.
\item \textbf{Execution:} A user attempted to disable the Windows Firewall via the command line interface on the Windows 11 agent.
\item \textbf{Result:} \textbf{Success.} By monitoring Process Creation (Event ID 4688), the system captured the exact command executed (\texttt{firewall off}). Custom Rule 100061 parsed the regex payload and generated a Level 12 Critical Alert, proving the configuration is version-agnostic and highly resilient.
\end{itemize}

\section{HAIT+ Compliance Coverage Analysis}
\label{sec:coverage-analysis}
Following the successful validation of the custom rules and deployment architecture, a comprehensive analysis of the SIEM's effectiveness against the 25 total HAIT+ cybersecurity indicators was conducted.

The engineered Wazuh implementation achieved a \textbf{60\% direct coverage rate (15 out of 25 indicators)}, drastically reducing the manual auditing workload for hospital IT staff. The coverage breakdown is as follows:
\begin{itemize}
\item \textbf{Supported by Default (32\% - 8 items):} Indicators met using Wazuh's native capabilities, such as standard authentication logs (\texttt{auth.log}) and File Integrity Monitoring (FIM).
\item \textbf{Standalone Configuration (24\% - 6 items):} Indicators met via the engineered Command-to-Rule pipeline, custom agent scripts, and hierarchical decoder overrides (e.g., Disk Space, CPU usage, Database Errors).
\item \textbf{Manager Correlation (4\% - 1 item):} Indicators met via manager-level aggregation rules, such as detecting mass device failures across the network.
\end{itemize}

The remaining \textbf{40\% (10 out of 25 indicators)} fall outside the scope of a Host-Based Intrusion Detection System (HIDS). Achieving 100\% compliance will require multi-system integration, including Network Intrusion Detection Systems (NIDS) for wire-level traffic analysis, and API integrations with organizational HR systems for contextual access control.
